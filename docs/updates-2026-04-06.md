# Athletica Backend Updates (2026-04-06)

This document summarizes the backend updates completed today.

## 1) Trainer Invite System Refactor

### Added models and lifecycle foundation

- Added `TrainerInviteCode` as the single permanent code per trainer:
  - fields: `id`, `trainerId` (unique), `code` (unique), `totalClients`.
- Added `TrainerClientInvite` as invite usage/history records:
  - fields include `trainerId`, `inviteCodeId`, `usedByClientId`, client metadata, `status`, timestamps.
  - status enum: `PENDING`, `ACCEPTED`, `EXPIRED`, `CANCELLED`.

### Invite responsibilities split by route

- New route for trainer code generation:
  - `POST /api/v1/trainer-invite-codes/`
- Invite operations route:
  - `GET /api/v1/client-invites/verify/:code`
  - `GET /api/v1/client-invites/`
  - `POST /api/v1/client-invites/approve/:clientId`

### Controller behavior changes

- `createTrainerClientInvite` now returns one reusable code per trainer.
- Code format is name-aware with prefix + sequence (starting from 100).
- `inviteLink` generation uses environment variables and appends `?code=<invite_code>`.
- `verifyTrainerClientInvite` validates code against `TrainerInviteCode`.
- `approveTrainerClientInviteByClientId`:
  - finds pending invite by `(trainerId, clientId)`
  - upserts into `TrainerClient`
  - sets relation status to `PAUSED` (not `ACTIVE`)
  - updates invite row to `ACCEPTED`
  - recalculates and persists `totalClients`
  - returns `trainerClient.id` in response.

## 2) Client Registration With Invite Code

- `clientRegister` in auth flow now supports optional invite code (`code` in body/query).
- If invite code is provided:
  - validates code in `TrainerInviteCode`
  - creates user + client profile
  - creates `TrainerClientInvite` history row with status `PENDING`
  - returns invite context in registration response.
- Registration message for invite path now indicates waiting for trainer approval.

## 3) Trainer-Client Status Lifecycle Alignment

- Approval step creates/updates the relation as `PAUSED` in `TrainerClient`.
- Existing trainer-client status endpoints remain available:
  - pause: `PATCH /api/v1/trainer-clients/updateStatusToPaused/:id`
  - activate: `PATCH /api/v1/trainer-clients/updateStatusToActive/:id`
  - end: `PATCH /api/v1/trainer-clients/updateStatusToEnded/:id`
- This supports a staged lifecycle: invite -> pending approval -> paused relation -> active later.

## 4) Intake Questionnaire Backend

### New domain support

- Added `ClientIntakeAnswer` model and `IntakeQuestionKey` enum.
- Added trainer-client-question unique constraint:
  - one answer per question per trainer-client pair.

### New endpoints

- `GET /api/v1/client-intake/questions`
- `POST /api/v1/client-intake/answers`
- `GET /api/v1/client-intake/answers/trainer/:trainerId`
- `GET /api/v1/client-intake/answers/client/:clientId`
- `GET /api/v1/client-intake/answers/trainer/:trainerId/client/:clientId`

### Behavior

- Answer upsert is idempotent by `(trainerId, clientId, questionKey)`.
- Access is role-scoped and validated with trainer-client link checks.

## 5) Prisma / Migration Updates

- Added migrations for:
  - intake enum and table
  - invite lifecycle evolution
  - dedicated invite code table
  - invite history indexing and constraints update.
- Final schema supports one trainer code + many invite history rows.

## 6) Environment and Config Notes

- `CLIENT_INVITE_LINK_BASE_URL` added in `.env.local` for invite-link generation.
- Server route mounting updated to include:
  - `client-intake`
  - `client-invites`
  - `trainer-invite-codes`

## 7) RBAC Seeding Stability

- `seedRbac.js` updated with deterministic role IDs.
- Existing roles are reused safely if present.
